## 三、目标用户画像

### 3.1 主要用户角色

#### 角色 1：数据工程师（小明）
- **背景**：制造业企业 IT 部门，3年工作经验
- **痛点**：
  - 数据源多且杂（ERP、MES、IoT、Excel）
  - 重复写 ETL 脚本，维护困难
  - 数据质量问题频繁
- **需求**：
  - 可视化数据集成工具
  - 自动化数据质量监控
  - 版本控制和协作

#### 角色 2：生产分析师（李工）
- **背景**：生产部门，熟悉业务但不懂编程
- **痛点**：
  - 想看数据需要找 IT 提需求
  - Excel 处理大数据卡顿
  - 无法实时监控生产异常
- **需求**：
  - 拖拽式分析界面
  - 自然语言查询
  - 实时告警通知

#### 角色 3：质量经理（王总）
- **背景**：质量管理部门负责人
- **痛点**：
  - 质量问题追溯困难
  - 无法预测质量风险
  - 报表制作耗时
- **需求**：
  - 质量预警模型
  - 一键生成报告
  - 根因分析工具

### 4.2 MVP 功能清单

#### P0 - 核心功能（必须）

| 模块 | 功能 | 用户故事 | 验收标准 |
|------|------|----------|----------|
| **数据集成** | 连接器管理 | 作为数据工程师，我可以配置数据源连接 | 支持 MySQL、PostgreSQL、Kafka、MQTT |
| | CDC 同步 | 我可以配置实时数据同步 | 数据延迟 < 5秒，支持断点续传 |
| | 数据转换 | 我可以编写转换逻辑 | 支持 SQL/Python 转换，可视化调试 |
| **数据存储** | 湖仓一体 | 数据自动存储到湖仓 | 支持 Iceberg 格式，时间旅行 |
| | 向量存储 | 支持文本向量化存储 | 集成 Milvus，支持语义检索 |
| **知识图谱** | 本体建模 | 我可以定义业务对象和关系 | 可视化建模界面，支持导入导出 |
| | 自动映射 | 自动将数据表映射到本体 | 准确率 > 80%，支持人工修正 |
| **AI 助手** | 自然语言查询 | 业务分析师可以用中文提问 | 支持 20+ 种常见业务问题类型 |
| | 智能报告 | 一键生成分析报告 | 自动选择图表类型，支持导出 PPT |
| **预警中心** | 规则配置 | 我可以配置业务规则 | 可视化规则编辑器，支持复杂条件 |
| | 实时告警 | 异常自动通知 | 支持微信/钉钉/邮件，延迟 < 1分钟 |

#### P1 - 重要功能（应该有）

| 模块 | 功能 | 说明 |
|------|------|------|
| **数据治理** | 数据血缘 | 自动追踪数据流向 |
| | 数据质量 | 内置质量规则，自动检测 |
| | 权限管理 | 行级/列级权限控制 |
| **行业模型** | 设备故障预测 | 基于振动、温度数据的预测模型 |
| | 质量根因分析 | 自动定位质量问题原因 |
| | 供应链优化 | 库存优化建议 |
| **协作功能** | 评论@ | 数据集上评论和@同事 |
| | 分享链接 | 生成可分享的洞察链接 |

#### P2 - 增强功能（可以有）

| 模块 | 功能 | 说明 |
|------|------|------|
| **移动端** | 移动 APP | 随时随地查看关键指标 |
| **开放 API** | REST API | 第三方系统集成 |
| **应用市场** | 插件生态 | 社区贡献的分析模板 |

## 五、非功能需求

### 5.1 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 查询响应时间 | P99 < 3秒 | 亿级数据量 |

## 七、技术要求

### 7.1 技术栈选型

| 层级 | 选型 | 理由 |
|------|------|------|
| **前端** | React + TypeScript + Tailwind | 生态成熟，团队熟悉 |
| **后端** | Python (FastAPI) + Go | Python AI 生态，Go 高性能 |
| **数据库** | PostgreSQL + ClickHouse + Milvus | OLTP + OLAP + 向量 |
| **流处理** | Apache Flink | 业界标准，中文社区活跃 |
| **图数据库** | Neo4j / NebulaGraph | 企业级，中文支持好 |
| **LLM** | vLLM + 国产大模型 | 私有化部署，合规 |
| **基础设施** | Kubernetes + Helm | 云原生，易于部署 |

### 7.2 架构原则

1. **模块化设计**：各组件可独立部署和扩展
2. **云原生优先**：容器化、微服务、DevOps
3. **开放标准**：支持开放 API 和标准协议
4. **安全内建**：从设计之初考虑安全
